# 44 — Product analytics plan

> **Type:** 🟡 Canonical for product analytics — **provider-neutral interface + dev-logger adapter, no vendor SDK yet** · **Last reviewed:** 2026-07-14
> **Status:** analytics is instrumented behind a provider-neutral interface and currently backed by a **dev-logger adapter** (`analyticsProvider: "dev-logger"`). No third-party analytics SDK is loaded and no data leaves the app until a provider is Approved (see §Open decisions). Nothing here approves a vendor, retention window, or consent model.

## Interface

Analytics is a small, provider-neutral interface (`apps/docs/lib/analytics.ts`): the app emits a whitelisted event name plus a small non-sensitive prop bag; an **adapter** decides what to do with it. Today the only adapter is the **dev-logger**, which logs events locally for development and verification. Swapping in a real provider is an adapter change — the call sites and the event whitelist do not change — and requires an Approved provider decision first.

Design rules:

- **Whitelist-only.** The app may emit only the events listed below. New events are added to the whitelist deliberately, not ad hoc.
- **No vendor SDK until Approved.** The dev-logger adapter is the default; no external script, network call, cookie, or fingerprint is introduced by analytics in the current posture.
- **Small non-sensitive props only.** Events carry short identifiers (e.g. component slug, pack slug, category) — never bodies, source, or personal data (see §Privacy).

## Conversion metrics (formulas — no current values)

Values are computed from the events above once real data exists. **No metric has a value in this document**; each defines a formula, its meaning, and the data required.

| Metric | Formula | Meaning | Data required |
|---|---|---|---|
| Homepage-to-catalog rate | `catalog_explore_clicked / homepage_viewed` | How well the homepage routes visitors into the catalog | both events with a shared session/visitor key |
| Catalog-to-component rate | `component_docs_viewed / (catalog_searched + category_filtered sessions)` | Whether browsing leads to inspecting a component | catalog-browse events + component-view events per session |
| Free-install copy rate | `free_install_copied / component_docs_viewed (Free)` | Intent to actually use a Free component | Free component views + copy events |
| Pro-page view rate | `pro_item_viewed / component_docs_viewed` | Share of evaluation that reaches Pro items | Pro views + all component views |
| Pack-page engagement | `pack_preview_interacted / pack_page_viewed` | Whether pack previews earn interaction | pack views + interaction events |
| Pack CTA rate | `pack_cta_clicked / pack_page_viewed` | Intent generated by a pack page | pack views + pack CTA clicks |
| Waitlist conversion | `waitlist_submitted / homepage_viewed` | Top-line demand capture in private-preview | homepage views + waitlist submissions |
| Checkout start | `checkout_started / access_cta_clicked` | Whether access intent turns into checkout | access CTA clicks + checkout starts |
| Checkout completion | `checkout_completed / checkout_started` | Checkout funnel efficiency | checkout start + complete events |
| Registry activation | `registry_item_installed / registry_item_requested` | Whether entitled requests turn into installs | registry request + install events |
| Pack interest | `pack_page_viewed / (pack_explore_clicked + catalog sessions)` | Relative pull of packs vs individual items | pack views + explore clicks |
| Category interest | distribution of `category_filtered` by category | Which workflow categories draw attention | category-filtered events with category prop |
| Free-to-Pro interest | `pro_cta_clicked / free_install_copied` | Whether Free usage creates Pro intent | Pro CTA clicks + Free copy events |

Notes: rates that span a session (e.g. homepage-to-catalog) require a **non-identifying** session/visitor key; that key is itself an Open decision (§Open decisions) and must respect the Privacy rules.

## Privacy

- **Never record** source code, secrets/tokens/keys, prompt/comment/request/response bodies, HTTP headers, or any personal data beyond an Approved policy.
- **Never** put query text, code, or personal data in event props — use derived, non-sensitive values (e.g. query *length*, not the query; slug, not content).
- Events carry only **small, non-sensitive props** (slugs, categories, entitlement ids, coarse counts, a source label).
- No cross-source personal-data compilation; no third-party identifiers; no fingerprinting in the current posture.
- Any session/visitor key must be non-identifying and is gated by the consent decision below.

## Open decisions

- **Which provider** replaces the dev-logger adapter (self-hosted vs vendor). — **Open**
- **Retention** window for stored events. — **Open**
- **Consent / cookie handling** (whether any provider requires consent, and the banner/decline model). — **Open**
- **Session/visitor keying** for cross-event funnel metrics (non-identifying scheme). — **Open**

Until a provider is Approved, analytics stays on the dev-logger adapter and no data leaves the app.
