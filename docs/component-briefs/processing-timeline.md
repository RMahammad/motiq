# Brief — Processing Timeline

- **Tier:** Pro · **Category:** File workflows
- **Problem:** After a file is ingested it usually runs through a *pipeline of processing stages* (scan → transcode → thumbnail → caption → publish). Apps need to show ONE item's journey through those stages — each stage's status, progress, output artifact, warning/error, and duration — but this is rebuilt ad hoc, inaccessibly, and often fakes progress.
- **Use case:** A media/asset processing panel the app drives (e.g. a product-video pipeline). The component presents the app-supplied stages for a single subject as an ordered rail, follows the current stage, surfaces per-stage output and logs, and emits intent callbacks. It never runs a stage, uploads, or advances `progress` itself.
- **Main states:** empty · in-flight (a current stage) · partial success (completed with warnings / some skipped) · failure · resolved. Per-stage: `pending · queued · active · paused · completed · warning · failed · skipped · cancelled` (icon + text, never colour alone).
- **Main interaction:** click/keyboard a stage header to expand its detail (output, logs, metadata, error, warning, timing); retry a failed stage; skip a skippable stage; cancel the whole job while in flight; restart once resolved; toggle vertical/horizontal layout and compact density.
- **Animation purpose:** communicate the subject's progression — current-stage pulse (active only, never after resolve), rail "travelled" segments, stage-detail expand/collapse, overall-progress bar. Not decoration. No faked per-stage progress, no per-frame React state, no per-% announcements, no indefinite animation on completed stages. Reduced motion renders final state, bars snap.
- **API sketch:** `<ProcessingTimeline stages title subtitle? jobStatus? currentStageId? layout? compact? activeStageId?/defaultActiveStageId?/onActiveStageChange? followCurrent? defaultExpanded? onRetryStage? onSkipStage? onCancel? onRestart? renderStageOutput? formatTimestamp? />`; stage = `{ id, label, description?, status, progress?, startTime?, endTime?, duration?, attempt?, error?, warning?, output?, logs?, metadata?, skippable? }`. Exports `ProcessingTimeline` + `ProcessingStage` + `ProcessingStatus` + `ProcessingTimelineProps`.
- **Accessibility:** stages are a semantic ordered list; the current stage carries `aria-current="step"` + a visible "Current stage" label; each stage header is a real disclosure `<button>` with `aria-expanded`; active/paused stages with app-supplied progress expose `role="progressbar"` + `aria-valuetext`; a failed stage's error is associated via `aria-describedby`; Retry/Skip/Cancel/Restart are ordinary keyboard buttons; status is always icon + text (never colour alone); a single polite `role="status"` region announces stage transitions only (never per-%); focus is kept on a stage header after Retry/Skip; reduced-motion safe.
- **Mobile behavior:** vertical rail stacks with large touch targets; the horizontal layout collapses to the vertical rail below `sm`; controls wrap; optional `compact` density.
- **Dependencies:** `motion` + `@motionkit/{utils,primitives}` (`useReducedMotion`, `useControllableState`, `statusVars`, `formatTimestamp`). No Node/`next`/Remotion imports.
- **Similarity concern:** distinct from **File Upload Pipeline** (one file's *upload* progress) and **Multi-file Queue** (managing a *collection* with priority lanes/concurrency). This tracks ONE item through *ordered post-ingest processing stages* with per-stage output/logs, partial success, and a duration summary. Conceptually adjacent to Agent Run Timeline (staged run) but a different domain (file/media processing, not agent tool-calls/reasoning/approvals), different composition (subject header, output artifacts, dual vertical/horizontal layout), and an original API. Generic pipeline pattern; clean-room implementation. Low–Moderate.
- **Release criteria:** ~5 targeted tests (current-stage focus + ordered progression, retry/skip/cancel callbacks, partial-success + failure state, keyboard stage expansion) + axe; rapid gate (desktop/mobile/light/dark/reduced/keyboard, typecheck, build, registry validates, clean-fixture install, no forbidden imports, no console/hydration errors).
