# Brief — Task Dependency Map

- **Tier:** Pro · **Category:** Productivity
- **Problem:** Dependency views ship as generic pan/zoom node graphs that are pointer-only, hide "what blocks this / what it unblocks", and re-lay-out unpredictably. Teams need to read blocked/blocking work fast without dragging.
- **Use case:** An **application-controlled** task-dependency visualization — tasks/statuses/scheduling/cycle validation/persistence supplied by the app; the component owns reading + light editing (select, inspect prerequisites/dependents, add/remove prerequisite, move group, collapse group, highlight path, present app-supplied cycle error). Not a PM app, not a graph editor.
- **States:** unselected · selected · blocked (with reason) · bottleneck · on-active-path · filtered-out (dimmed) · group-collapsed · optimistic-move-pending · move-failed (rollback) · cycle-error · empty.
- **Interaction:** select task (click/keyboard), roving-tabindex arrow navigation across dependency levels + within a lane, navigate relationships from the detail panel, add/remove prerequisite (menu + remove button), move task to another group (optimistic + rollback), collapse/expand group, filter by status (built-in chips), highlight path (`activePath`), map ⇄ compact list.
- **Layout:** deterministic, computed in plain JS — dependency-depth columns (longest path, cycle-guarded) inside group lanes; dependency lines are **static SVG** (never continuously animated) drawn from computed coordinates (no measurement → SSR/test-stable). **Not** a pan/zoom node graph.
- **API sketch:** `<TaskDependencyMap tasks selectedTaskId onSelectedTaskChange onAddDependency(taskId,depId) onRemoveDependency(taskId,depId) onMoveTask(taskId,toGroup) layout groups activePath cycleError compact renderTask renderDetails />`; task = { id, title, status, priority?, assignee?, startDate?, dueDate?, progress?, dependencyIds, blockedReason?, group?, milestone?, metadata? }; status = planned|ready|active|blocked|completed|cancelled.
- **Accessibility:** **dragging never required** — full keyboard selection + relationship navigation; SVG lines are decorative (aria-hidden), relationships conveyed in the textual detail/list; status = icon + text (never colour alone); polite announcements on selection/move/rollback; focus preserved across edits and restored after rollback; compact **list view = mobile dependency-detail mode**.
- **Animation purpose:** subtle select emphasis + edge highlight transitions + detail-panel reveal; reduced-motion removes all transitions and draw. No continuous line animation.
- **Dependencies:** motion + `@motionstack/{utils,primitives}` (`useReducedMotion`, `useControllableState`, `useOptimisticAction`, `statusVars`) **only — no external graph/drag library** (clean-room).
- **Similarity concern:** dependency views exist, but the lane+depth layered layout, keyboard-first reading model, app-owned cycle presentation, and optimistic group-move are original + clean-room. Medium — kept original (no copied APIs/effects/layouts).
- **Performance:** pure-JS layout memoized on tasks; nodes are `React.memo`'d; edges recompute only on selection/path change; no per-frame work.
- **Release criteria:** ~6 targeted tests (keyboard select + relationship navigation, add/remove dependency callbacks, app-supplied cycle-error presentation, optimistic move rollback, list-view fallback) + axe; rapid gate; Pro → protected registry routing.
